## 4. 탄소규제 맥락 인사이트 3개

### 인사이트 1 — CBAM 확정 시행이 VoC 최대 동인이며, 신고 실무 문의가 상위 공급망에 집중·일부 고위험
- **내용**: CBAM 관련 VoC가 단일 규제 주제 중 최다이며, EU 세관 반려·신고 마감 초과 등 계약 위험으로 직결되는 긴급 건이 상위 고객(OEM·1차 협력사)에서 발생.
- **근거**: CBAM 관련 VoC **10건**(001·006·010·016·023·025·032·036·041·045, 44건의 22.7%) 중 **9건이 OEM·1차 협력사**(036만 공공 조달 문의). 이 중 010(EU 세관 보고서 반려)·045(신고 마감 초과, 수출계약 위험)는 긴급·고위험. 최초 신고기한 문의(001)가 동일 내용으로 재유입(031 중복)될 만큼 반복 = 미해결 잔존 신호.
- **비즈니스 의미**: 2026년 1월 CBAM 확정 시행으로 신고 실무(기한·양식·비용·내재량) 문의가 급증하는 국면. **신고 기한 알림 + EU 양식 사전 검증** 기능이 없으면 고가치 OEM 이탈로 직결. 신고 실무 FAQ·가이드를 선제 배포해 CS 부하를 분산해야 함.

### 인사이트 2 — 협력사 1차 데이터 수집 병목이 최대 미충족 니즈(Scope3·CBAM 공통), 자동화 ROI는 이미 입증됨
- **내용**: "1차 협력사·OEM이 하위 협력사에서 탄소 데이터를 수집"하는 과정의 병목이 기능 요청의 핵심 축. 반면 자동화가 도입된 고객의 만족도는 급상승.
- **근거**: 협력사 데이터 수집·자동화 요청 008(협력사 200개 수동입력)·018(일괄 발송)·023(CBAM 협력사 공문 템플릿)와 Scope3 계산 지원 005·033. 반대 축으로 자동화 도입 후 만족한 034(업무시간 절반 단축)·039(감축 지점 파악) — ROI 실증.
- **비즈니스 의미**: CSRD의 Scope 3(공급망) 데이터 제공 요구와 CBAM 내재량 산정이 겹치며 "협력사 데이터 수집"이 공통 통증점. **협력사 온보딩·일괄 수집 자동화** 확대가 최우선 기능 투자처이며, 034 성공사례를 세일즈 레퍼런스로 전환 가능.

### 인사이트 3 — 계산 정확도·검증 가능성 요구가 반복, 법적 보고(CSRD·CBAM)에서 신뢰 = 계약유지 리스크
- **내용**: 계산 결과에 대한 불신·검증 요구가 반복 등장. 다만 정확도 자체는 개선 체감도 존재하므로, 핵심 리스크는 "정확도 악화"가 아니라 "**검증 가능성(근거 공개·제3자 연계) 부족**".
- **근거**: 수치 불신·검증 요구 014(외부기관 20% 차이)·033(경쟁사 30% 차이)·019(GWP/IPCC 미반영)·002(Scope2 급변)와 제3자 검증 연계 문의 015·040. 반례로 024(집계 오류 감소 만족)처럼 개선 체감도 존재.
- **비즈니스 의미**: CSRD 제3자 검증(Limited Assurance)과 CBAM 내재량 보고는 법적 효력을 가지므로 수치 근거가 불투명하면 = 법적 리스크·신뢰 훼손. **계산 로직·배출계수 출처 표시, GWP(IPCC) 자동 업데이트, ISO 14064·인증기관 양식 연계**가 신뢰 방어의 핵심.

## [Standard] 파이프라인 재현성 — 새 CSV로 동일 리포트 자동 생성

새 VoC CSV를 넣으면 정제 → 분류 → 집계 → 리포트를 **동일 품질로 재생성**하도록 Claude Code Skill 4종으로 패키징했다. (위치: `.claude/skills/{analyze,insight,generate,review}/SKILL.md`)

### 사용법 (Claude Code 터미널)
1. 새 `voc.csv`를 `data/` 폴더에 교체
2. `/analyze` — 컬럼·결측·중복·날짜형식 점검 + 정제 계획 제안
3. `/insight` — 4유형 분류(CTA 규칙) + 부가 신호·다중질문·긴급 태깅 → `output/voc-classification.csv` 저장 + 인사이트 3개 도출
4. `/generate` — 분류표·유형별 집계·인사이트·주간 리포트를 `template.md` 형식으로 생성(`voc-report.md` + `voc-report.html`)
5. `/review` — 채점 기준 대조 자가 점검 후 제출

### 무엇이 재현성을 보장하나
- **판정 기준을 스킬에 고정**: 정제 규칙(중복 제거·날짜 ISO·결측 유지), 분류 규칙(명시적 요청/질문 CTA 기준 + 경계 케이스 타이브레이커), 인사이트 프레임, 점검 체크리스트를 스킬 문서에 내장 → 데이터가 바뀌어도 사람이 판단을 다시 정하지 않는다.
- **숫자는 매번 재집계**: 리포트의 건수·비율은 `voc-classification.csv`에서 다시 계산(하드코딩 금지) → 새 데이터로 자동 갱신.
- **어조가 아닌 CTA 기준**: 감정·긴급도가 아니라 "명시적으로 요구한 행동"으로 판정하므로, 독립적으로 다시 분류해도 결과가 흔들리지 않는다(재현성 검증에서 44건 동일 결과).

- **경영·전략** → "규제 환경이 어떻게 움직이나?" (선제 대응 신호)

## [Challenge] 제품 개선 기획안

VoC 패턴에서 제품 개선 기획안 2건을 도출하고, **우선순위 판단 기준과 기획안 양식을 직접 설계**했다. (상세 원본: 리포지토리 `기획안.md`) 아래 표·카드의 **시각화 버전은 `output/voc-report.html` 하단 '제품 개선 기획안' 섹션**에 기존 리포트와 동일 디자인 시스템으로 수록했다(우선순위 3축 배점 카드 · 전체 후보 랭킹표 · 기획안 카드 2건 · 점수 막대 바, 이모지 없음).

### 우선순위 판단 기준

3축 100점 루브릭. 각 축을 기획안 목적과 1:1로 연결했다.

| 축 | 배점 | 무엇을 측정 | 연결되는 목적 |
|----|:---:|------------|--------------|
| **축1 누적 반복 빈도** | 40 | 동일 문제가 몇 건 반복되는가 = 미해결 GAP의 폭 | 제품 로드맵 근거 |
| **축2 부가 신호 강도** | 30 | 불만·부정 표현·긴급·재유입 = 이탈 위험의 세기 | 고객 이탈 예방 / 규제 조기 감지 |
| **축3 거래처 매출 규모** | 30 | 누가 제기하는가 = 위협받는 ARR 규모 | 고객 이탈 예방 |

**채점 규칙 (재현 가능)**
- **축1 누적 반복 빈도(40)**: 테마 매핑 VoC 수 — 1건 10 / 2건 20 / 3건 30 / 4건 이상 40
- **축2 부가 신호 강도(30 상한)**: VoC별 최고 신호 1개 — `긴급 10` / `불만 8` / `secondary 불만·명시적 부정 표현 5` / `순수 요청·문의 0`, (가산) 미해결 재유입·중복 재발 +6
- **축3 거래처 매출 규모(30 상한)**: 유형별 배점 차등 — `OEM 10` / `1차 협력사 7` / `2차 협력사 4` / `기타 2`
- **티어**: P0 ≥ 75 / P1 60–74 / P2 < 60
- **선정 규칙(단일 실행성)**: 기획안은 단일하게 실행 가능한 개선 단위여야 함. 이질적 기능이 다수 묶인 광범위 테마는 빈도축이 과대 계상되므로 분해 후 로드맵으로 이관.

### 기획안 1 — CBAM 신고·제출 실무 완결성 · **P0 (96점)**

- **문제 정의**: 카본링크는 CBAM 보고서를 *생성*해 주지만, 이후 **신고 기한 관리 → EU 양식 적합성 검증 → 세관 제출·수리**는 고객 몫이다. 2026년 1월 CBAM 확정 시행으로 실무 부담이 급증하며, 상위 고객(OEM·1차)이 **마감을 놓치거나 EU 세관에서 양식 문제로 반려**되는 사후 케이스가 반복 — 수출 계약 위험 = 최고가치 고객 이탈 위험으로 직결.
- **근거 VoC (3건 + 클러스터)**:
  - `id 010` (OEM·불만·**긴급**) "CBAM 보고서가 EU 세관에서 반려됐습니다 … 긴급합니다." → 생성물–EU 양식 불일치.
  - `id 045` (OEM·문의·**긴급**) "신고 마감일이 지났는데 … 수출 계약이 위험합니다." → 기한 관리 부재.
  - `id 001/031` (1차·문의·**재유입**) "CBAM 신고 기한이 정확히 언제인가요?" → 031로 동일 재유입 = 미해결 잔존.
  - (클러스터) 016 구매의무·비용 / 025 내재량 제출형식 / 041 단가추산 / 032 권한이전 / 036 공공조달 — CBAM 실무 반복 수요.
- **개선 제안**: ① CBAM 신고 캘린더 + `D-30/7/1` 자동 알림 ② 제출 전 **EU 최신 양식 사전 검증**(반려 예방) ③ `생성→검증→제출→수리` 상태 추적 보드 ④ EU-ETS 연동 인증서 비용 추산기.
- **우선순위 (96점 · P0)**: 축1 40(실무 VoC 9건) · 축2 26(긴급 010·045 각 10 + 001 재유입 +6) · 축3 30(OEM 4·1차 4·기타 1 → 상한). **판단 근거**: 단일 규제 주제 중 최다 반복 + 재유입으로 구조적 GAP 입증(로드맵), 긴급·계약위험 신호가 전부 OEM(이탈 예방·매출 규모), 유일하게 긴급 2건 → **로드맵 내 최우선 착수**.

### 기획안 2 — 공급망 Scope3 협력사 데이터 수집 자동화 · **P0 (81점)**

- **문제 정의**: CSRD Scope 3 제공 의무와 CBAM 내재량 산정이 겹치며 OEM·1차가 **수백 개 하위 협력사 데이터**를 수집해야 한다. 현재는 **수동 입력/개별 링크 발송**이라 협력사 200개 기준 주당 수십 시간 소요 + 미응답 시 데이터 누락. 반대로 자동화 도입 고객은 **업무시간 절반 단축**(ROI 실증).
- **근거 VoC (3건 + ROI 반례)**:
  - `id 008` (OEM·불만) "협력사가 200개인데 일일이 손으로 입력 … 시간이 너무 걸립니다." → 대량 수동입력 병목.
  - `id 018` (기타·기능요청) "일괄 발송 기능이 필요합니다 … 개별 링크 전송이 너무 번거롭습니다." → 일괄 발송 부재.
  - `id 023` (1차·기능요청) "협력사들이 협조를 안 합니다. 공문·요청 템플릿을 제공해 주실 수 있나요?" → 수집 독려 수단 부재.
  - `id 034` (2차·칭찬, **ROI 반례**) "수집 자동화 도입 이후 업무 시간이 절반 이상 줄었습니다." → 효과 실증 = 세일즈 레퍼런스.
- **개선 제안**: ① 협력사 리스트 업로드 → 데이터 요청 **일괄 발송** ② 협력사용 **무계정 간이 입력 폼** ③ 응답률·미응답 **실시간 추적 + 자동 리마인드** ④ 공문·요청 템플릿 내장.
- **우선순위 (81점 · P0)**: 축1 40(008·018·023·034) · 축2 18(008 불만 8 + 018·023 부정표현 5 + 034 칭찬 0) · 축3 23(OEM+1차+2차+기타). **판단 근거**: 같은 병목을 여러 각도가 반복 지목(로드맵), 최상위 OEM 불만이나 긴급 없음 → 축2가 기획안1보다 낮아(18<26) **차순위**, 유일하게 개선 ROI가 실증(034)돼 회수 리스크 최저.

> **검증 메모**: 두 기획안이 인용한 모든 VoC id의 `customer_type`·`primary_type`·원문 인용을 `voc-classification.csv`·`data/voc.csv`와 대조 → 날조·오귀속 0건. 6개 테마 재매핑 재채점으로 랭킹 재현성 확인, 상위 2건은 경계 판정(023 이중 귀속 등)에도 불변. 023은 산출물이 데이터 수집 기능이므로 기획안2에 귀속(이중계상 회피).
