## [2026-07-01] 결정 1 — 분류 스키마: 하이브리드(Primary + Secondary + 다중질문)
- 결정 내용: 각 VoC에 배타적 **Primary 유형 1개**(집계·비율의 유일 기준)를 부여하고, 그 외 감지된 부차 신호는 **Secondary aspect 태그**(집계 미반영, 인사이트·기획안 근거용)로, 서로 다른 질문이 2개 이상이면 **다중질문 플래그**로 별도 기록.
- 선택한 이유: 채점표가 "4유형 분류표 + 유형별 건수·비율(합계 100%)"을 단일 라벨 표로 요구(CLAUDE.md L27-28)하므로 순수 multi-label은 합계 무결성이 깨져 감점 위험. Primary 단일 집계로 규격을 100% 충족하면서, Secondary로 경계 케이스의 정보 손실을 막아 Standard·Challenge 근거로 재사용.
- 대안으로 고려했던 것: (i) 순수 multi-label(합계 100% 규격과 충돌, 기각), (iii) aspect-unit 분해 후 라우팅(표 2개 병존·재현성 방어 부담으로 과설계, 기각).

## [2026-07-01] 결정 2 — Primary 판정 규칙: "명시적 요청/질문(CTA) 기준"
- 결정 내용: Primary는 고객의 **명시적 요구 행동(CTA)**으로 판정. 감사·만족=칭찬 / 기능·자료·서비스의 추가·제공·개선을 명시 요청=기능 요청 / 결함·오류·미흡 표출 또는 결함 해결 요구=불만 / 정보·기준·방법·가능여부 질문=일반 문의. **어조(tone)·긴급도는 Primary를 바꾸지 않고** Secondary/urgency로 기록.
- 선택한 이유: 어조 기반 분류는 판정자마다 흔들려 재현성이 낮음. CTA 기준은 규칙이 명확해 독립 블라인드 재분류에서 44건 전부 동일 결과가 나옴(재현성 입증).
- 대안으로 고려했던 것: 리스크/이탈 신호 우선(불만 상향) 방식 — 샘플 HTML 스타일. 신뢰 훼손 어조를 불만으로 올리지만 CTA와 어긋나 일관성 저하로 기각.

## [2026-07-01] 결정 3 — "불편·부재 호소"의 기능요청 vs 불만 우선순위 (008·038)
- 결정 내용: 기존 기능의 불편이나 기능 부재를 호소하되 **명시적 요청 동사(추가/제공/개선/만들어)가 없으면 Primary=불만**, 잠재 기능 니즈는 **Secondary=기능요청**으로 보존. 요청 동사가 있으면 Primary=기능 요청.
- 선택한 이유: 규칙 조항 "기능 부재로 인한 니즈(기능요청)"와 "요청 동사 없는 불편 호소(불만)"가 008·038에서 충돌 → CTA 유무를 최종 타이브레이커로 명문화(감사관 권고 반영).
- 대안으로 고려했던 것: 잠재 니즈를 기능요청 Primary로 승격 — 불만 신호가 묻혀 CS 대응 우선순위에서 누락되므로 기각.

## [2026-07-01] 결정 4 — 014·033·045 = 일반 문의 (샘플과 의도적 차이)
- 결정 내용: 수치 불신(014 20% 차이·033 30% 차이)과 긴급 케이스(045 마감 초과)는 명시적 요구가 각각 "근거/로직 설명", "연장 여부·문의 방법"이므로 **Primary=일반 문의**, secondary=불만/urgency로 기록.
- 선택한 이유: 결정 2(CTA 기준)의 일관된 적용. 샘플 HTML은 이를 불만으로 분류했으나, 어조·긴급도로 유형을 바꾸지 않는 원칙을 우선.
- 대안으로 고려했던 것: 샘플대로 불만 분류. 규칙 일관성이 깨지고 002·030 등과 처리가 어긋나 기각(대신 Secondary로 불만 신호 보존).

## [2026-07-01] 결정 5 — 데이터 정제
- 결정 내용: (a) id 031 = id 001 완전 동일 중복 → **제거, 44건 기준 집계**. (b) 날짜 3형식(2026-05-02 / 2026/05/06 / 5월 4일) → **ISO(YYYY-MM-DD) 통일**. (c) 채널 결측 2건(032·035) → **결측 유지**(임의 추정 금지).
- 선택한 이유: 중복은 집계 왜곡을 유발하므로 제거가 정확. 채널은 내용 분류에 불필요하므로 추정보다 결측 명시가 데이터 무결성에 안전.
- 대안으로 고려했던 것: 031을 '동일 이슈 반복 재유입'으로 별도 카운트 — 완전 동일 행이라 데이터 아티팩트로 판단, 대신 인사이트 1에서 '미해결 잔존 신호'로만 정성 활용.

## [2026-07-01] 결정 6 — problem.md의 "id 043 유형 오입력" 항목은 무효 처리
- 결정 내용: problem.md L30의 "id 043은 불만인데 칭찬으로 오입력" 과제는 **성립하지 않음**으로 판단. 실제 CSV에 `유형` 컬럼이 없어(설계 중 제거) 교정 대상이 없으며, 043 내용은 "기능 개선 요청드립니다"라는 명시 CTA가 있어 **기능 요청**으로 분류.
- 선택한 이유: design-conversation.md 기준 `유형` 컬럼 제거가 확정이므로 problem.md 문구는 구버전 잔재. CTA 기준 규칙과도 일치.
- 대안으로 고려했던 것: 불만으로 분류(problem.md 서술 존중) — CTA 기준·실데이터와 불일치로 기각(불만은 Secondary로 보존).

## [2026-07-01] 결정 7 — 다중 질문 플래그: 016·025·040
- 결정 내용: 서로 독립적으로 답변해야 할 질문이 2개 이상인 016(구매의무+비용추산)·025(제출형식+표준양식)·040(검증요건+ICMA 연계)에 **다중질문 플래그**. 유형은 모두 일반 문의로 동일하므로 4유형 분포·집계에는 영향 없음. CS 워크로드·FAQ 설계 지표로만 활용.
- 선택한 이유: 다중 질문은 응대 건수(워크로드)를 좌우하나 유형 분포를 왜곡하면 안 되므로 별도 레이어로 분리(적대 검증 반영: 045는 단일 의도로 반려, 025는 누락분 추가).
- 대안으로 고려했던 것: 질문 단위로 행을 분해(unit 집계) — 표가 이원화되어 채점 규격과 마찰, 기각.

## [2026-07-01] 결정 8 — 인사이트 3개 선정 기준·근거
- 결정 내용: 44건에서 도출 가능한 여러 패턴 중 리포트 인사이트를 **① CBAM 확정 시행이 VoC 최대 동인(신고 실무·고위험 상위 고객 집중) ② 협력사 1차 데이터 수집 병목(Scope3·CBAM 공통 통증점, 자동화 ROI 실증) ③ 계산 정확도가 아닌 '검증 가능성' 요구 반복(법적 보고 신뢰=계약 리스크)** 3개로 선정. 선정은 3필터 통과분만 승격: **(a) prevalence**(반복·건수 규모) **(b) 비즈니스 임팩트**(계약·이탈·법적 리스크 직결) **(c) 탄소규제 맥락 연결성**(CSRD·CBAM). 단순 최다 빈도가 아니라 "규제 맥락 × 대응 액션 전환 가능성"을 우선.
- 선택한 이유: 채점표가 "단순 수치 나열 아닌 비즈니스 해석"을 인사이트 배점 조건으로 명시(CLAUDE.md L29). 규제와 연결되고 리뉴어스랩의 구체 대응(기능 투자·FAQ·검증 연계)이 도출되는 패턴만 인사이트로 승격하고, 액션이 안 나오는 단순 분포는 배제해 30점 요건에 정렬.
- 대안으로 고려했던 것: (i) 유형 분포(일반 문의 38.6% 최다)를 그대로 인사이트화 — 단순 수치 나열이라 감점 리스크로 기각(대신 §2 집계로만 제시). (ii) 채널·고객 세그먼트 분포 — 규제 맥락 연결이 약해 후순위로 기각.

## [2026-07-01] 결정 9 — 우선순위 프레임워크(3축 100점) 설계 + 단일 실행성 선정 규칙
- 결정 내용: Challenge 기획안 우선순위를 **본인 설계 3축 100점 루브릭**으로 채점 — **축1 누적 반복 빈도(40)** / **축2 부가 신호 강도(30 상한: 긴급10·불만8·부정표현/secondary불만5·순수요청0, 재유입+6)** / **축3 거래처 매출 규모(30 상한: OEM10·1차7·2차4·기타2)**. 티어 **P0≥75 / P1 60–74 / P2<60**. 여기에 **단일 실행성(coherence) 선정 규칙**을 추가: 이질적 기능이 다수 묶여 빈도축이 과대 계상된 테마는 분해 후 로드맵 이관. → 2위 '보고서·프레임워크 통합'(90점, 이질 기능 7건)을 이 규칙으로 제외(분해 시 각 ≤64점)하고, 단일 실행 단위 상위 2건 **CBAM 신고·제출(96)·공급망 Scope3 수집 자동화(81)**를 기획안으로 채택.
- 선택한 이유: 빈도만으로는 "많다=중요" 함정에 빠지므로 이탈 위험(축2)·매출 규모(축3)를 균등 배분해 "반복되면서, 화가 났고, 큰 고객이 겪는" 문제를 최상단으로 끌어올림. 각 축을 기획안 목적(로드맵 근거/이탈 예방/규제 조기 감지)과 **1:1로 연결**해 채점의 설명력·재현성을 확보(채점 10점 '본인 설계+근거' 요건). 총점 단순 정렬만 쓰면 번들 테마가 인위적 1위가 되어 실행 불가능한 기획안이 채택되므로 coherence 규칙으로 교정.
- 대안으로 고려했던 것: (i) 단순 반복 빈도순 정렬 — 번들 테마 과대평가·단일 실행성 무시로 기각. (ii) 정성적 임팩트 판단만으로 선정 — 판정자 편차로 재현성이 낮아 기각(정량 루브릭으로 대체). (iii) 2위(90점) 그대로 채택 — 이질 기능 7건 묶음이라 단일 기획안으로 실행 불가, 로드맵 에픽('리포팅 2.0')으로 이관.
