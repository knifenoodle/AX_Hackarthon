# /review — 제출 전 자가 점검

산출물이 채점 기준을 충족하는지 항목별로 확인하고, **미흡 항목은 수정 지시까지** 제시한다.

## 검증 절차
1. **집계 무결성**: `output/voc-classification.csv`의 primary_type 4종 건수 합계 = 정제 후 총건수인가? 비율 합계 = 100%(반올림 오차 ±0.1 허용)인가?
2. **리포트 일치**: voc-report.md / voc-report.html의 건수·비율·표 행 수가 CSV와 정확히 일치하는가? (하드코딩으로 어긋난 곳 없는지 대조)
3. **규칙 정합성(재현성)**: 각 행의 primary_type이 /insight의 CTA 규칙과 rationale에 부합하는가? 어조·긴급도로 유형이 바뀐 행은 없는가? 경계 케이스(불편 호소·수치 불신·긴급 문의)가 규칙대로 처리됐는가?
4. **인사이트 품질**: 3개 모두 (a) CSRD/CBAM 맥락 연결, (b) 실제 집계에서 인용한 근거 숫자·id, (c) 비즈니스 행동 제안을 갖췄는가? 단순 수치 나열이 아닌가?
5. **형식**: template.md 섹션 구조(개요·분류표·집계·인사이트·의사결정 로그)를 갖췄는가? 노션/구글독스에 붙여넣기 가능한 마크다운인가?
6. **재현성(Standard)**: 새 CSV로 /analyze→/insight→/generate 재실행 시 동일 품질이 나오는가? 하드코딩된 숫자·id에 의존하지 않는가?
7. **주간 리포트(Standard)**: KPI·CS 즉시대응·제품 반복요청·규제 패턴 섹션과 WoW 증감이 있는가? 설계 근거(CS·제품팀 관점)가 서술됐는가?
8. **의사결정 로그**: 경계 케이스·정제·예외 처리가 decisions.md에 근거와 함께 기록됐는가?

## 채점 기준 대조 체크리스트 (CLAUDE.md)
| 항목 | 배점 | 확인 |
|------|------|------|
| 분류 정확도 (4유형 + 경계 케이스 근거) | 40 | ☐ |
| 건수·비율 집계 (합계 100%) | 20 | ☐ |
| 탄소규제 인사이트 3개 (비즈니스 해석) | 30 | ☐ |
| 제출 형식 (공개 링크) | 10 | ☐ |
| [Standard] 파이프라인 재현성 | 15 | ☐ |
| [Standard] 주간 리포트 설계 + 근거 | 15 | ☐ |
| [Challenge] 기획안 실효성 | 20 | ☐ |
| [Challenge] 양식·우선순위 설계 | 10 | ☐ |

## 출력 형식
- 각 항목 **통과 / 미흡**과 근거를 한 줄로.
- 미흡 항목은 구체적 수정 지시(어느 파일 어느 부분을 어떻게)로 마무리.
- 숫자 검산 결과(유형별 건수, 합계, 비율)를 표로 첨부한다.
