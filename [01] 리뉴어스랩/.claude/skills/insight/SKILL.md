# /insight — 분류 기준 적용 + 인사이트 도출

`/analyze`의 정제 계획을 반영해 각 VoC를 분류하고, 탄소규제 맥락 인사이트를 도출한다.
결과는 `output/voc-classification.csv`로 저장한다(이후 `/generate`가 이 파일을 읽어 리포트를 만든다).

## 분류 스키마 (하이브리드)
행마다 다음을 부여한다.
- **primary_type** (집계·비율의 유일 기준, 배타적 1개): `불만` / `기능 요청` / `칭찬` / `일반 문의`
- **secondary_aspects** (참고용, 집계 미반영): 한 건에 함께 담긴 다른 신호(예: `불만·정확도`, `기능·대량입력`). 없으면 공란.
- **multi_question** (`Y`/`N`): 서로 독립적으로 답해야 할 질문이 2개 이상이면 `Y`.
- **urgency** (`높음`/공란): 업무 마비·계약 위험·법정 기한 임박 등 즉시 대응 신호. **유형을 바꾸지 않는다.**

## Primary 판정 규칙 — "명시적 요청/질문(CTA) 기준"
고객이 **명시적으로 요구한 행동(Call To Action)**으로 유형을 정한다. 어조·감정·긴급도는 primary를 바꾸지 않고 secondary/urgency로만 기록한다.

| 유형 | 판정 신호 (CTA) |
|------|-----------------|
| **칭찬** | 감사·만족·긍정 피드백 표현 (요구 없음) |
| **기능 요청** | 기능·자료·서비스의 **추가/제공/개선/생성**을 명시 요청 ("추가해주세요", "가이드 만들어 달라", "있으면 좋겠다") |
| **불만** | 결함·오류·미흡·불편·부재를 표출하거나, 결함 해결을 요구 |
| **일반 문의** | 정보·기준·방법·가능여부를 **질문**(설명·확인 요구) |

### 경계 케이스 타이브레이커 (재현성 핵심)
1. **불편·부재 호소 vs 기능 요청**: 기존 기능의 불편이나 기능 부재를 말하되 **요청 동사(추가/제공/개선/만들어)가 없으면 → 불만**. 잠재 기능 니즈는 `secondary=기능요청`으로 보존. 요청 동사가 있으면 → 기능 요청.
2. **수치 불신·급변 vs 불만**: 계산 결과를 의심하거나 급변에 불안해하지만 명시 요청이 "**근거/로직 설명**"이면 → 일반 문의(`secondary=불만·정확도`). 어조가 강해도 유형은 바꾸지 않는다.
3. **긴급/마감 초과 문의**: 긴급하더라도 명시 요청이 "**연장 여부·문의 방법**" 같은 질문이면 → 일반 문의 + `urgency=높음`.
4. **결함 신고 + 개선 요청 동시**: 결함이 핵심이면 → 불만(`secondary=기능요청`). 개선 요청이 핵심 CTA면 → 기능 요청(`secondary=불만`).

> 이 규칙은 판정자에 따라 흔들리지 않도록 어조가 아닌 CTA에 고정되어 있다. 동일 데이터를 독립 재분류해도 같은 결과가 나오는 것이 목표다.

## 절차
1. `/analyze`의 정제 규칙을 적용한다(중복 제거, 날짜 ISO, 결측 유지, 빈 내용 제외).
2. 각 행을 위 규칙으로 분류하고, `content_summary`(30자 내외)와 `rationale`(판정 근거 한 줄)을 적는다.
3. **다중질문 판정**: 하나의 VoC 안에 답변 단위가 다른 질문이 2개 이상이면 `multi_question=Y`. (유형 분포에는 영향 없음 — 워크로드 지표로만 사용)
4. `output/voc-classification.csv`로 저장. 컬럼: `id,date,channel,customer_type,primary_type,secondary_aspects,multi_question,urgency,content_summary,rationale`
5. **자가 검산**: 4유형 건수 합계 = 분모(정제 후 건수)인지 확인. 어긋나면 원인을 찾고 재검토.
6. **인사이트 3개 도출** (아래 프레임).

## 인사이트 도출 프레임 (CSRD·CBAM 맥락)
"단순 수치 나열"이 아니라 **패턴 → 근거 → 비즈니스 의미**로 해석한다. 각 인사이트는:
- **내용**: 규제(CSRD/CBAM/Scope3/검증 등) 맥락과 연결된 패턴 한 문장.
- **근거**: 관련 VoC 건수·대표 `id`·반례까지. (숫자는 실제 집계에서 인용)
- **비즈니스 의미**: 리뉴어스랩(카본링크)이 취해야 할 행동(제품·CS·세일즈).

권장 인사이트 축(데이터에 따라 조정):
1. **규제 시행 임박이 만든 VoC 급증**(CBAM 신고 실무·기한·양식·비용) + 고가치 고객(OEM·1차) 집중·이탈 위험.
2. **공급망 데이터 수집 병목**(CSRD Scope3 + CBAM 내재량 공통) — 기능 요청의 핵심 축, 자동화 ROI 근거.
3. **계산 정확도·검증 가능성 신뢰 이슈** — 법적 보고에서 근거 투명성 = 계약 유지 리스크.

용어·세그먼트는 `context/company-info.md`, 규제 동향은 `context/industry-news.md`를 근거로 사용한다.
