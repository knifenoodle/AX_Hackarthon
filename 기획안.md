# challenge 기획안 과제
- 내용 : 분류된 VoC에서 도출한 제품 개선 기획안 샘플 1—2건 (문제 정의 / 근거 VoC(2-3건) / 개선 제안 / 우선순위)
- 우선순위 판단 기준을 본인이 직접 설계 -> 판단 근거 설명 포함
- 데이터 출처 : `[01] 리뉴어스랩/output/voc-classification.csv` (원본 45건 → 중복 031 제거, **44건** 기준) · 원문 `data/voc.csv`

# 기획안 목적
- **제품 로드맵 근거**: 요청 VoC 빈도 → 기능 개발 우선순위 데이터 기반 결정
- **고객 이탈 예방**: 불만 VoC 처리 속도 → 계약 갱신율(NRR) 직결
- **규제 변화 조기 감지**: 특정 키워드 문의 급증 → 규제 업데이트 콘텐츠 선제 대응

# 우선순위 판단 기준 (본인 설계)

## 판단 규칙 (설계 시드)
- 누적 반복 빈도, 부가 신호(불만, 부정적 표현과 유형), 거래처 유형별 매출 규모(OEM > 1차 협력사 > 2차 협력사 > 기타) 에 가중치부여(각 유형 속에서 배점 상이하게)

## 설계 철학 — 왜 이 3축인가 (기획안 목적과 1:1 연결)
| 축 | 배점 | 무엇을 측정 | 연결되는 목적 |
|----|:---:|------------|--------------|
| **축1 누적 반복 빈도** | 40 | 동일 문제가 몇 건 반복되는가 = 미해결 GAP의 폭 | 제품 로드맵 근거 |
| **축2 부가 신호 강도** | 30 | 불만·부정 표현·긴급·재유입 = 이탈 위험의 세기 | 고객 이탈 예방 / 규제 변화 조기 감지 |
| **축3 거래처 매출 규모** | 30 | 누가 제기하는가 = 위협받는 ARR 규모 | 고객 이탈 예방 |

> 빈도(1차 신호)에 가장 높은 배점을 주되, 빈도만으로는 "많다=중요"의 함정에 빠지므로 **이탈 위험(축2)과 매출 규모(축3)**를 균등 배분해 "반복되면서, 화가 났고, 큰 고객이 겪는" 문제를 최상단으로 끌어올린다.

## 채점 규칙 (100점 만점 · 재현 가능)

**축1 — 누적 반복 빈도 (40점)**
테마에 매핑되는 VoC 수(primary + 관련 secondary 잠재 니즈 포함). `1건=10 / 2건=20 / 3건=30 / 4건 이상=40`.

**축2 — 부가 신호 강도 (30점 상한)**
테마 VoC별 위험 신호 점수의 합. 한 VoC는 **가장 높은 1개 신호만** 채택:
- `urgency=높음` → **10점**
- `primary_type=불만` → **8점**
- `secondary 불만 태그` 또는 원문의 **명시적 부정 표현**(불편·번거로움·비효율·협조 안 함·막막 등) → **5점**
- 순수 요청·문의(신호 없음) → **0점**
- (가산) 동일 문의 **미해결 재유입/중복 재발**(예: 001↔031) → **+6점**

**축3 — 거래처 매출 규모 (30점 상한) — "유형별 배점 차등"**
테마 VoC별 거래처 유형 배점의 합(매출 기여 순):

| 거래처 유형 | 건당 배점 | 근거 |
|------------|:---:|------|
| OEM | **10** | 글로벌 공급망 앵커 계정 — 이탈 시 ARR 손실 최대 |
| 1차 협력사 | **7** | OEM 직납, 다수 하위 협력사 견인 |
| 2차 협력사 | **4** | 300여 곳, 개별 규모는 작음 |
| 기타 | **2** | 금융·공공·유통 등 |

**총점 = 축1 + 축2 + 축3** → 티어: **P0 ≥ 75 / P1 60–74 / P2 < 60**

## 선정 규칙 — 단일 실행성(coherence)
기획안은 **단일하게 실행 가능한 개선 단위**여야 한다. 이질적 기능이 다수 묶인 광범위 테마는 점수가 인위적으로 부풀려지므로(빈도축 과대 계상), **분해 후 개별 평가**하고 로드맵으로 이관한다. → 아래 랭킹에서 `보고서·프레임워크 통합(90)`이 이 규칙에 해당(사유는 랭킹표 각주).

## 전체 후보 랭킹 (44건 → 6개 테마 재매핑 채점)
| 순위 | 테마 | 축1 | 축2 | 축3 | **총점** | 티어 | 채택 |
|:---:|------|:---:|:---:|:---:|:---:|:---:|:---:|
| 1 | CBAM 규제 신고·제출 실무 완결성 | 40 | 26 | 30 | **96** | P0 | **기획안 1** |
| 2 | 보고서·프레임워크 통합 유연성 | 40 | 20 | 30 | 90 | P0 | 로드맵 이관¹ |
| 3 | 공급망 Scope3 데이터 수집 자동화 | 40 | 18 | 23 | **81** | P0 | **기획안 2** |
| 4 | 계산 정확도·산정 로직 투명성 | 40 | 23 | 14 | 77 | P0 | 후속 3순위² |
| 5 | 플랫폼 안정성·시스템 연동 | 30 | 28 | 13 | 71 | P1 | 상시 CS SLA |
| 6 | 규제·방법론 인플랫폼 가이드 | 30 | 15 | 11 | 56 | P2 | 콘텐츠 백로그 |

> ¹ 2위(90)는 CSRD 이중중요성·TCFD 통합·차트·크레딧 표시·분기 자동생성·SBTi·커스터마이징 등 **이질적 기능 7건**의 묶음. 단일 기획안 부적합 → 하위 테마로 분해 시 각 ≤64점(P1/P2). '리포팅 2.0' 에픽으로 로드맵 이관.
> ² 4위 계산 정확도(77)는 코히런트하나 OEM·1차 앵커가 0건이라 매출축(14)이 구조적으로 낮음 → 신뢰 리스크 관점에서 후속 우선.

# 기획안1

## 내용

### 제목 — CBAM 신고·제출 실무 완결성 (기한 알림 + 반려 예방 + 제출 추적)  ·  **우선순위 P0 (96점)**

### 문제 정의
카본링크는 CBAM 보고서를 **생성**해 주지만, 생성 이후의 **신고 기한 관리 → EU 양식 적합성 검증 → 세관 제출·수리**까지는 고객이 알아서 처리한다. 2026년 1월 CBAM 확정 시행으로 실무 부담이 급증한 국면에서, 상위 고객(OEM·1차 협력사)이 **기한을 놓치거나(마감 초과)**, **EU 세관에서 양식 문제로 반려**되는 사후 대응 케이스가 반복 발생 중이다. 이는 곧 **수출 계약 위험 = 최고가치 고객 이탈 위험**으로 직결된다.

### 근거 VoC (3건 + 클러스터)
- **id 010** (OEM · 불만 · **긴급**) : *"카본링크에서 출력한 CBAM 보고서가 EU 세관에서 반려됐습니다. 양식 문제인지 수치 문제인지 확인 부탁드립니다. 긴급합니다."* → 생성물이 EU 양식과 불일치 → 반려.
- **id 045** (OEM · 문의 · **긴급**) : *"CBAM 신고 마감일이 지났는데 연장 신청이 가능한지 … 수출 계약이 위험합니다. 긴급 대응이 필요합니다."* → 기한 관리 부재로 마감 초과.
- **id 001 / 031** (1차 협력사 · 문의 · **동일 문의 재유입**) : *"CBAM 신고 양식 제출 기한이 정확히 언제인가요? 올해 처음 신고라 헷갈립니다."* → 동일 내용이 031로 재유입 = **미해결 잔존 신호**.
- (실무 문의 클러스터) 016 구매의무·비용 / 025 내재량 제출 형식 / 041 인증서 단가 추산 / 032 담당자 권한이전 / 036 공공조달 영향 — CBAM 실무 전반의 반복 수요.

### 개선 제안
1. **CBAM 신고 캘린더 + 자동 알림** : 분기·연간 마감일을 플랫폼에 내장, `D-30 / D-7 / D-1` 이메일·배너 알림 (045·001 재발 차단).
2. **EU 양식 사전 검증(반려 예방)** : 제출 전 EU 최신 서식·필수항목·수치 범위를 자동 대조, 불일치 항목을 사전 경고 (010 반려 차단).
3. **제출 상태 추적** : `생성 → 검증 → 제출 → 수리` 4단계 상태보드로 미제출·반려 건 가시화.
4. **인증서 비용 추산기** : EU-ETS 단가 연동 예상 비용 산출 (016·041 문의 흡수).

### 우선순위 산출 (96점 · P0)
| 축 | 산출 | 점수 |
|----|------|:---:|
| 축1 반복 빈도 | CBAM 실무 VoC 9건(001·006·010·016·025·032·036·041·045) → 4건 이상 상한 | **40** |
| 축2 부가 신호 | 010 긴급 10 + 045 긴급 10 + 001 재유입 +6 = 26 | **26** |
| 축3 매출 규모 | OEM 4건(10×4)+1차 4건(7×4)+기타 1건(2) = 원합 70 → 상한 30 | **30** |
| | **총점** | **96 / P0** |

## 판단 근거
- **반복 빈도(제품 로드맵 근거)** : 단일 규제 주제 중 **최다 반복**(9건). 특히 001은 031로 **재유입**되어 "묻고도 해소되지 않은" 구조적 GAP임을 입증 → 로드맵 최우선 편입 정당.
- **부가 신호(이탈 예방·조기 감지)** : 긴급 2건(010·045)이 모두 **계약 위험** 문구 동반. 규제 마감이라는 외생 트리거가 만든 신호이므로 방치 시 이탈로 직결.
- **매출 규모(이탈 예방)** : 긴급·고위험 건이 전부 **OEM**에서 발생. 앵커 계정 이탈은 하위 협력사 연쇄 이탈로 확산 → 매출축 상한 도달이 곧 위협 규모의 크기.
- **즉시성** : 유일하게 긴급(urgency) 신호가 2건 붙은 P0 → 로드맵 내에서도 **가장 먼저 착수**해야 하는 건.
- **검증** : 인용 9개 id의 거래처유형·원문 인용을 원본 CSV와 대조해 **날조·오귀속 없음** 확인. 001 재유입 가산은 프레임워크가 명시 허용한 규칙.

# 기획안2

## 내용

### 제목 — 공급망 Scope3 협력사 데이터 수집 자동화  ·  **우선순위 P0 (81점)**

### 문제 정의
CSRD의 Scope 3(공급망) 데이터 제공 의무와 CBAM 내재량 산정이 겹치면서, OEM·1차 협력사가 **수백 개 하위 협력사로부터 탄소 데이터를 수집**해야 한다. 현재 카본링크는 이를 **개별 수동 입력 / 협력사별 개별 링크 발송**으로 처리해 협력사 200개 기준 주당 수십 시간이 소요되고, **협력사 미응답 시 데이터 누락**이 발생한다. 신고 기한이 다가올수록 누락 리스크가 커지는 구조다. 반대로 자동화가 도입된 고객은 **업무시간 절반 단축**을 체감(ROI 실증).

### 근거 VoC (3건 + ROI 반례)
- **id 008** (OEM · 불만) : *"공급망 2차 협력사 데이터 입력 기능이 너무 불편합니다. 협력사가 200개인데 일일이 손으로 입력해야 해서 시간이 너무 걸립니다."* → 대량 수동입력 병목(최상위 고객).
- **id 018** (기타 · 기능 요청) : *"공급업체 탄소 데이터 공유 요청을 일괄 발송하는 기능이 필요합니다. … 개별 링크 전송이 너무 번거롭습니다."* → 일괄 발송 부재.
- **id 023** (1차 협력사 · 기능 요청) : *"협력사들이 협조를 안 합니다. 공문이나 요청 템플릿 양식을 제공해 주실 수 있나요?"* → 수집 독려 수단(템플릿) 부재.
- **id 034** (2차 협력사 · 칭찬, **ROI 반례**) : *"수집 자동화가 도입된 이후로 업무 시간이 절반 이상 줄었습니다."* → 개선 효과 실증 = 세일즈 레퍼런스.

### 개선 제안
1. **협력사 리스트 업로드 → 요청 일괄 발송** : CSV 업로드 한 번으로 전 협력사에 데이터 요청 이메일 자동 발송 (008·018 해소).
2. **협력사용 무계정 간이 입력 폼** : 협력사가 카본링크 계정 없이 링크로 데이터 직접 입력.
3. **응답 현황 실시간 추적 + 자동 리마인드** : 응답률(%)·미응답 목록 대시보드, 미응답 자동 독촉 (023의 "협조 안 함" 해소).
4. **공문·요청 템플릿 내장** : 협력사 협조 요청 표준 문서 제공.

### 우선순위 산출 (81점 · P0)
| 축 | 산출 | 점수 |
|----|------|:---:|
| 축1 반복 빈도 | 008·018·023·034 = 4건 → 4건 이상 상한 | **40** |
| 축2 부가 신호 | 008 불만 8 + 018 부정표현 5 + 023 부정표현 5 + 034 칭찬 0 = 18 | **18** |
| 축3 매출 규모 | OEM 10 + 기타 2 + 1차 7 + 2차 4 = 23 (상한 미만) | **23** |
| | **총점** | **81 / P0** |

## 판단 근거
- **반복 빈도(제품 로드맵 근거)** : 대량입력·일괄발송·수집 독려가 서로 다른 각도에서 **같은 병목(협력사 데이터 수집)**을 반복 지목 → 명확한 로드맵 신호.
- **부가 신호(이탈 예방)** : 최상위 고객(008 OEM)이 **불만**으로 표출, 018·023은 "번거롭다·협조 안 함"의 부정 표현. 긴급은 없어 기획안1보다 축2가 낮음(18 < 26) → **차순위** 근거.
- **매출 규모** : OEM(008)·1차(023)가 포함돼 매출축 23점 확보. 다만 OEM·1차 밀도가 CBAM 테마보다 낮아 상한 미도달 → 총점이 96에 못 미치는 이유.
- **전략 가치(가점 요인)** : 유일하게 **개선 ROI가 실증된**(034 업무 50% 단축) 테마. 채점에는 반영 안 되나(칭찬=신호 0) 세일즈 레퍼런스·근거로 병기 → 투자 대비 회수 리스크가 가장 낮음.
- **검증** : 인용 4개 id의 거래처유형·원문 전건 일치, 신호 채택 규칙 정확(적대 검증 통과).

- **재현성** : 6개 테마를 44건에 재매핑해 동일 규칙으로 재채점 → 랭킹 sound 확인. 채택 상위 2건은 어떤 경계 판정(예: 023의 CBAM/공급망 이중 귀속, 030의 테마 귀속)에도 **1·2위가 뒤집히지 않음**.
- **근거 무결성** : 두 기획안이 인용한 모든 VoC id의 `customer_type`·`primary_type`·원문 인용을 `voc-classification.csv`·`data/voc.csv`와 대조 → **날조·오귀속 0건**.
- **경계 처리** : 023은 "CBAM 데이터 + 협력사 수집"의 이중 성격이나, 요청 산출물(공문/일괄수집 템플릿)이 데이터 수집 기능이므로 **기획안2에 귀속**(기획안1과 이중계상 회피). 이 경우 기획안1은 96점 유지.
- **후속 로드맵** : ③ 계산 정확도·산정 로직 투명성(77, P0) → ④ 리포팅 2.0 에픽(2위 90점 테마 분해) → 플랫폼 안정성(71)은 상시 CS SLA로 관리.
