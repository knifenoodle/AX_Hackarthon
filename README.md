## 문제 정의

리뉴어스랩은 공급망 탄소 데이터 자동화 플랫폼 **카본링크(CarbonLink)** 를 운영하며 현대차·BMW·테슬라 등 글로벌 OEM 공급망 협력사 310여 곳을 고객으로 둔 딥테크 스타트업이다.

탄소규제 컨설팅 특성상 고객 VoC는 이메일·채널톡·설문·미팅 등 다양한 채널로 **산발적으로 유입**되는데, 현재 CS팀은 이를 **수작업 엑셀로 정리**하고 있어 유형별 패턴 파악이 어렵고 비즈니스 의사결정에 활용하지 못한다. 특히 **CBAM 확정 시행(2026년 1월)·CSRD 적용 확대**로 문의·불만이 급증하는 국면이다.

> 흩어진 VoC를 **구조화해 제품·CS·전략 의사결정의 근거로 전환**하는 것 — 이것이 이 과제의 목표다.

4. **(Challenge) 제품 개선 기획안 2건** — 문제 정의·근거 VoC·개선 제안·우선순위

## 재현 파이프라인 (Standard)

새 `voc.csv`를 넣으면 **정제 → 분류 → 집계 → 리포트**를 동일 품질로 재생성하도록 Claude Code Skill 4종으로 패키징했습니다. 판정 기준(정제 규칙·CTA 분류 규칙·인사이트 프레임·점검 체크리스트)을 스킬 문서에 고정해, 데이터가 바뀌어도 사람이 판단을 다시 정하지 않습니다.

```
data/voc.csv 교체
   │
   ├─ /analyze   컬럼·결측·중복·날짜형식 점검 + 정제 계획 제안 (데이터 수정 X, 진단만)
   ├─ /insight   4유형 분류(CTA 기준) + 부가 신호·다중질문·긴급 태깅 → voc-classification.csv 저장 + 인사이트 3개
   ├─ /generate  분류표·집계·인사이트·주간 리포트 생성 → voc-report.md + voc-report.html
   └─ /review    채점 기준 대조 자가 점검 (통과/미흡 체크리스트)
```

**재현성을 보장하는 3가지 원칙**
- **판정 기준을 스킬에 고정** — 데이터가 바뀌어도 규칙을 다시 정하지 않음
- **숫자는 매번 재집계** — 건수·비율을 `voc-classification.csv`에서 다시 계산(하드코딩 금지)
- **어조가 아닌 CTA 기준** — 감정·긴급도가 아니라 "명시적으로 요구한 행동"으로 판정 → 독립 재분류에서 44건 완전 일치

## 방법론 핵심

### 분류 스키마 — 하이브리드 (대표 유형 + 부가 신호)

각 VoC에 **배타적 대표 유형 1개**(집계·비율의 유일 기준)를 부여하고, 그 외 감지된 신호는 **부가 신호**(집계 미반영, 인사이트·기획안 근거용)로, 서로 다른 질문 2개 이상은 **다중질문 플래그**로 별도 기록합니다.

> 순수 multi-label은 "합계 100%" 규격과 충돌 → 대표 유형 단일 집계로 규격을 충족하면서, 부가 신호로 경계 케이스의 정보 손실을 방지.

### 유형 판정 규칙 — 명시적 요청/질문(CTA) 기준

| 유형 | 판정 기준 |
|------|-----------|
| 칭찬 | 감사·만족 표현 |
| 기능 요청 | 기능·자료·서비스의 추가·제공·개선을 **명시 요청** |
| 불만 | 결함·오류·미흡 표출 또는 결함 해결 요구 |
| 일반 문의 | 정보·기준·방법·가능 여부 질문 |

**어조·긴급도는 유형을 바꾸지 않고** 부가 신호/긴급도로 기록 → 판정자 편차 제거, 재현성 확보.

### 우선순위 판단 기준 — 3축 100점 루브릭 (직접 설계)

| 축 | 배점 | 무엇을 측정 | 연결 목적 |
|----|:---:|------------|-----------|
| **축1 누적 반복 빈도** | 40 | 동일 문제 반복 건수 = 미해결 GAP의 폭 | 제품 로드맵 근거 |
| **축2 부가 신호 강도** | 30 | 불만·부정 표현·긴급·재유입 = 이탈 위험 세기 | 고객 이탈 예방 / 규제 조기 감지 |
| **축3 거래처 매출 규모** | 30 | 누가 제기하는가 = 위협받는 ARR 규모 | 고객 이탈 예방 |

- **채점**: 축1 `1건10/2건20/3건30/4건+40` · 축2 `긴급10·불만8·부정표현5·순수요청0` (+재유입 +6) · 축3 `OEM10·1차7·2차4·기타2`
- **티어**: P0 ≥ 75 / P1 60–74 / P2 < 60
- **단일 실행성(coherence) 규칙**: 이질적 기능이 다수 묶여 빈도축이 과대 계상된 테마는 분해 후 로드맵 이관 → 실행 불가능한 번들 테마가 인위적 1위가 되는 것을 방지
